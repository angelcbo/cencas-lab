# Module: Tenant

**Package:** `com.cenicast.lis.tenant`

**What it does:** Manages the lifecycle of tenants (labs) in the platform. A tenant represents one clinical lab customer — it's the root of all data isolation.

**Who uses it:** **SUPER_ADMIN only.** Every endpoint in this module requires `hasRole('SUPER_ADMIN')`. Lab users (LAB_ADMIN and below) have no access to this module.

**Endpoints:** `GET/POST/PUT/DELETE /api/v1/tenants/...` and `POST /api/v1/tenants/{tenantId}/users`

---

## What Is a Tenant?

A **tenant** is one clinical lab that subscribes to the platform. Each tenant:
- Has a unique `slug` (URL-safe identifier, e.g., `demo-lab`) — immutable after creation
- Has a `name` (display name, e.g., `"Demo Lab"`)
- Has a `timezone` (IANA, e.g., `"America/Mexico_City"`) — used for display only; all data is UTC
- Has a `taxRate` (e.g., `0.1600` for 16% IVA) — applied to all invoices
- Has an `active` flag — `false` when the lab's subscription ends (soft delete)

Every piece of patient data, order, result, and billing record in the system is owned by exactly one tenant.

## Tenant Lifecycle

### Create a tenant — `POST /api/v1/tenants`

**Service:** `TenantService.createTenant(TenantRequest req, UserPrincipal actor, String ipAddress)`

1. Check slug uniqueness: `tenantRepository.existsBySlug(req.slug())` — throws `ApiException(409)` if taken
2. Create the `Tenant` entity:
   - `slug` = from request (immutable after save)
   - `timezone` = from request, or `"America/Mexico_City"` if null
   - `taxRate` = from request, or `0.1600` if null
   - `active = true`
3. Save to database
4. Audit: `AuditService.recordResource(CREATE_TENANT, actor, "Tenant", tenant.id, {slug: ...}, ipAddress)`
5. Return `TenantResponse`

**Returns:** `201 Created` with the new tenant in the body, `Location` header pointing to the new resource.

### Read tenants

- `GET /api/v1/tenants` — paginated list of all tenants (`?page=0&size=20&sort=createdAt,desc`)
- `GET /api/v1/tenants/{id}` — get one tenant by UUID (404 if not found)

### Update a tenant — `PUT /api/v1/tenants/{id}`

Only `name`, `timezone`, and `taxRate` can be updated. **The slug is immutable** — it's not in `TenantRequest`'s accepted fields. Changing a slug after creation could break URLs and integrations.

The service does a partial update: if `req.timezone()` is null, the existing timezone is preserved.

**Audit:** `AuditService.recordResource(UPDATE_TENANT, actor, "Tenant", tenant.id, {slug: ...}, ipAddress)`

### Deactivate a tenant — `DELETE /api/v1/tenants/{id}`

**Soft delete only — no data is ever deleted.** Sets `active = false`.

Why soft delete?
- Regulations may require data retention (patient records in Mexico must be kept for years)
- An accidentally deactivated tenant can be reactivated
- The data remains auditable

**Audit:** `UPDATE_TENANT` event with `{slug: ..., action: "deactivated"}` in metadata.

## Creating the First LAB_ADMIN — `POST /api/v1/tenants/{tenantId}/users`

This special endpoint lets SUPER_ADMIN create the first administrator for a new tenant.

**Why this endpoint exists separately from `POST /api/v1/users`?**

The regular `POST /api/v1/users` endpoint requires `hasRole('LAB_ADMIN')` — only a tenant's own admin can create users in that tenant. SUPER_ADMIN doesn't have the `LAB_ADMIN` role, so they can't use that endpoint. This `POST /api/v1/tenants/{tenantId}/users` endpoint serves as the bootstrap mechanism, exclusively for SUPER_ADMIN.

**How the tenantId is determined:**
- From the **path variable** `{tenantId}` — NOT from the JWT claim (SUPER_ADMIN has no `tenantId` in their token)
- The `UserService.createUser()` method receives `tenantId` explicitly as a parameter

**What it does:**
- Validates the request body (`CreateUserRequest` — same as regular user creation)
- Sets `tenantId` on the new user from the path variable
- Hashes the password with BCrypt(12)
- Saves the user
- Records `CREATE_USER` audit event

## `Tenant` Entity

**File:** `tenant/model/Tenant.java`

| Field | Column | Type | Notes |
|---|---|---|---|
| `id` | `id` | UUID | PK, generated by Java `UUID.randomUUID()` in `@PrePersist` |
| `slug` | `slug` | VARCHAR(63) | Unique, immutable identifier (e.g., `demo-lab`) |
| `name` | `name` | VARCHAR(255) | Display name |
| `timezone` | `timezone` | VARCHAR(50) | IANA timezone string |
| `taxRate` | `tax_rate` | NUMERIC(5,4) | `0.1600` = 16% IVA |
| `active` | `active` | boolean | `false` = deactivated (soft deleted) |
| `createdAt` | `created_at` | TIMESTAMPTZ | Set in `@PrePersist`, never updated |
| `updatedAt` | `updated_at` | TIMESTAMPTZ | Set in `@PrePersist`, updated in `@PreUpdate` |

**Why is `Tenant` NOT extended from `TenantAwareEntity`?**
`TenantAwareEntity` is for entities that are *scoped to* a tenant (they have `tenant_id` pointing to some other tenant). `Tenant` IS the tenant — it makes no sense to filter the tenants table by tenant ID. It's the root entity.

## Tenant Onboarding Runbook

Step-by-step to onboard a new lab:

```bash
# 1. Log in as SUPER_ADMIN
TOKEN=$(curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@cenicast.com","password":"ChangeMe123!"}' \
  | jq -r '.accessToken')

# 2. Create the tenant
TENANT_ID=$(curl -s -X POST http://localhost:8080/api/v1/tenants \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"slug":"clinica-guadalajara","name":"Clínica Guadalajara","taxRate":0.1600}' \
  | jq -r '.id')
echo "Tenant ID: $TENANT_ID"

# 3. Create the first LAB_ADMIN for this tenant
curl -s -X POST http://localhost:8080/api/v1/tenants/$TENANT_ID/users \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@clinica-guadalajara.com",
    "password": "SecurePass123!",
    "firstName": "Carlos",
    "lastName": "Martínez",
    "role": "LAB_ADMIN"
  }' | jq .

# 4. Give the LAB_ADMIN their credentials. They log in at:
# POST /api/v1/auth/login with tenantSlug: "clinica-guadalajara"
```
