# Module: Audit & Common Utilities

**Package:** `com.cenicast.lis.common`

**What it does:** Provides the cross-cutting infrastructure shared by all modules — audit logging, error handling, correlation IDs, and security utilities. These are not a "feature" that end users interact with directly; they are the foundation every other module depends on.

This document covers:
- Audit logging (`common/audit/`)
- Exception handling (`common/exception/`)
- Request correlation (`common/util/CorrelationIdFilter`)

---

## Why Audit Logging Matters

Every meaningful action in the system — logins, failed logins, user creation, result validation, payment recording — is written to an immutable `audit_events` table. This serves three purposes:

1. **Compliance** — Clinical labs in Mexico must be able to demonstrate who accessed or changed patient data and when (health data retention requirements).
2. **Security forensics** — If a breach occurs, audit logs show what happened, when, and from which IP.
3. **Debugging** — When something goes wrong in production, the audit trail often shows the sequence of events that led to the problem.

The `audit_events` table is **append-only**: rows are inserted but never updated or deleted. There is no `@PreUpdate` on `AuditEvent`. Once written, a row is permanent.

## `AuditEvent` Entity

**File:** `common/audit/AuditEvent.java`

| Field | Column | Type | Notes |
|---|---|---|---|
| `id` | `id` | UUID | PK, generated by `UUID.randomUUID()` in `@PrePersist` |
| `tenantId` | `tenant_id` | UUID (nullable) | Null for SUPER_ADMIN actions (they have no tenant) |
| `actorId` | `actor_id` | UUID | Who performed the action. No FK to `users` table (intentional — see below) |
| `actorEmail` | `actor_email` | VARCHAR(255) | Denormalized email for human-readable logs |
| `action` | `action` | VARCHAR(100) | The action string, e.g., `"LOGIN"`, `"CREATE_PATIENT"` |
| `resourceType` | `resource_type` | VARCHAR(100) (nullable) | What kind of resource was affected, e.g., `"User"`, `"Tenant"` |
| `resourceId` | `resource_id` | UUID (nullable) | The UUID of the affected resource |
| `metadata` | `metadata` | JSONB (nullable) | Structured context. NEVER contains PHI — see PHI Rule below |
| `ipAddress` | `ip_address` | VARCHAR(45) (nullable) | Client IP at the time of the action |
| `correlationId` | `correlation_id` | VARCHAR(36) (nullable) | The `X-Correlation-Id` of the request — links audit row to logs |
| `createdAt` | `created_at` | TIMESTAMPTZ | Set in `@PrePersist`; never modified |

**Why no FK on `actor_id`?**

`audit_events.actor_id` stores the user's UUID but does NOT declare a `FOREIGN KEY REFERENCES users(id)`. This is intentional. If a user account is deactivated or eventually hard-deleted, we don't want the audit rows to break or cascade-delete. The audit history must survive the user record. Denormalizing the email as `actor_email` means the logs remain readable even without the user row.

**Why no `@PreUpdate`?**

Audit rows must be immutable. Declaring no `@PreUpdate` hook means Hibernate will still update the row if code explicitly calls `setField()` followed by `save()`, but since `AuditService` only ever calls `save()` on a newly constructed entity, this is never triggered. The lack of `@PreUpdate` is a signal to future developers: "this entity is not meant to be updated."

**The `@ColumnTransformer(write = "?::jsonb")` on `metadata`:**

```java
@Column(columnDefinition = "jsonb")
@ColumnTransformer(write = "?::jsonb")
private String metadata;
```

PostgreSQL's `jsonb` type requires the value to be cast explicitly when inserting a `String` via JDBC. The `?::jsonb` tells PostgreSQL to cast the bound parameter string to `jsonb` at insert time. Without this annotation, PostgreSQL would reject the insert with a type mismatch error.

## `AuditService`

**File:** `common/audit/AuditService.java`

`AuditService` provides three methods. All three use `Propagation.REQUIRES_NEW`.

### `record()` — core method

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void record(
        AuditAction action,
        UUID actorId,
        String actorEmail,
        UUID tenantId,
        String resourceType,
        UUID resourceId,
        Map<String, Object> metadata,
        String ipAddress)
```

Builds an `AuditEvent`, serializes `metadata` to JSON using `ObjectMapper`, picks up the current `correlationId` from MDC (`MDC.get("correlationId")`), and calls `auditEventRepository.save(event)`.

### `recordAuth()` — convenience overload for auth events

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void recordAuth(AuditAction action, UUID actorId, String email, UUID tenantId, String ip)
```

Calls `record()` with `resourceType = null`, `resourceId = null`, `metadata = null`. Used for: `LOGIN`, `FAILED_LOGIN`, `LOGOUT`, `TOKEN_REFRESH`, `TOKEN_REVOKED`, `PASSWORD_CHANGE`.

### `recordResource()` — convenience overload for CRUD events

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void recordResource(AuditAction action, UserPrincipal actor,
                           String resourceType, UUID resourceId,
                           Map<String, Object> meta, String ip)
```

Accepts a `UserPrincipal` instead of separate `actorId`/`email`/`tenantId` parameters. Used for: `CREATE_USER`, `UPDATE_USER`, `CREATE_TENANT`, `UPDATE_TENANT`, and all future resource events.

### Why `REQUIRES_NEW` propagation?

> Reference: [Spring Transaction Propagation](https://docs.spring.io/spring-framework/reference/data-access/transaction/declarative/tx-propagation.html)

`REQUIRES_NEW` means: "always start a new, separate transaction for this method — suspend the calling transaction if one exists."

The critical use case is `FAILED_LOGIN`:

1. `AuthService.login()` runs in a `@Transactional` method
2. The password doesn't match — we want to record a `FAILED_LOGIN` audit event
3. Then we throw `ApiException(401, "Invalid credentials")`
4. Spring sees the `RuntimeException` and marks the outer transaction for rollback
5. Without `REQUIRES_NEW`, the audit insert would be rolled back too — the `FAILED_LOGIN` would be lost
6. With `REQUIRES_NEW`, the audit insert commits in its own independent transaction before the outer transaction rolls back

This guarantee holds for all security events: failed logins, token revocations, and logouts always persist in the audit log even if the surrounding operation fails.

## PHI Rule — What Goes in Metadata

**PHI** (Protected Health Information) is any data that could identify a patient or reveal their medical condition. In Mexico, health data is regulated under **NOM-024-SSA3** and related standards. The principle is similar to HIPAA in the US: you must limit access to and storage of patient-identifying data.

**The rule for `audit_events.metadata` is strict and non-negotiable:**

### FORBIDDEN in metadata (never log these)

- Patient names, dates of birth, CURP (national ID), phone numbers, email addresses
- Lab result values (numeric or text) — e.g., glucose `5.4 mmol/L`, HIV `"reactive"`
- Diagnostic impressions, diagnoses, clinical notes
- Passwords, tokens, secrets of any kind
- Any field that could uniquely identify a patient's health status

### ALLOWED in metadata (log these freely)

- IDs (UUIDs) — e.g., `{"patientId": "a1b2c3..."}` — IDs alone cannot reconstruct PHI
- Status transitions — e.g., `{"status": "COMPLETED", "previousStatus": "IN_PROGRESS"}`
- Role names — e.g., `{"role": "LAB_ANALYST"}`
- Counts and aggregates — e.g., `{"itemCount": 3}`
- Tenant slugs — e.g., `{"slug": "demo-lab"}`
- Token family IDs (for breach detection)
- Action context that doesn't reveal PHI — e.g., `{"action": "deactivated"}`

### Why this matters in practice

When Sprint 5 adds result entry, the `ENTER_RESULT` audit event should record:

```json
{"orderId": "uuid", "orderItemId": "uuid", "studyCode": "GLU"}
```

NOT:

```json
{"patientName": "Juan García", "glucoseValue": "126 mg/dL", "flag": "H"}
```

The first tells us *that* a result was entered and *which order* it belongs to. The second exposes a patient's lab result in the audit log, which is likely accessible to SUPER_ADMIN and potentially other admins — a PHI violation.

## Correlation IDs and `CorrelationIdFilter`

**File:** `common/util/CorrelationIdFilter.java`

Every HTTP request in the system gets a **correlation ID** — a UUID that uniquely identifies that request. The ID appears in:
- The `X-Correlation-Id` response header
- The `correlationId` field of error response bodies
- Every log line emitted during that request (via MDC)
- The `correlation_id` column of any audit rows created during that request

This makes it trivial to trace a single request through all logs and audit rows.

### What is MDC?

> Reference: [SLF4J MDC documentation](https://www.slf4j.org/manual.html#mdc)

MDC (Mapped Diagnostic Context) is a thread-local map of key-value pairs maintained by the logging framework (SLF4J/Logback). Any key-value pair placed in MDC appears automatically in every log line emitted on that thread, without you having to pass it as a parameter.

For example, if you add the following to your Logback pattern:
```
%X{correlationId}
```

Every log line will automatically include the correlation ID for that request.

### How `CorrelationIdFilter` works

```java
@Override
protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
                                FilterChain filterChain) throws ServletException, IOException {
    String correlationId = request.getHeader("X-Correlation-Id");
    if (correlationId == null || correlationId.isBlank()) {
        correlationId = UUID.randomUUID().toString();  // generate if not provided
    }
    MDC.put("correlationId", correlationId);          // make available to all logs
    response.setHeader("X-Correlation-Id", correlationId);  // echo back in response
    try {
        filterChain.doFilter(request, response);
    } finally {
        MDC.remove("correlationId");  // clean up — thread will be reused
    }
}
```

Key points:
- **Inbound:** If the client sends `X-Correlation-Id: <uuid>`, that ID is used (useful for client-side tracing)
- **Generated:** If no ID is provided, a new UUID is generated automatically
- **Echo:** The ID is always set on the response header so the client can reference it when reporting issues
- **Cleanup:** `MDC.remove()` in the `finally` block is critical — web server threads are reused; forgetting to clean up means the next request on this thread would carry the previous request's correlation ID

### Using correlation IDs in practice

When a user reports a bug, ask them for the `X-Correlation-Id` from the response header. Then search logs:

```bash
docker logs cenicast-backend 2>&1 | grep "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
```

All log lines from that request will appear, in order.

## `GlobalExceptionHandler`

**File:** `common/exception/GlobalExceptionHandler.java`

`GlobalExceptionHandler` is annotated with `@RestControllerAdvice` — a Spring mechanism that intercepts exceptions thrown from any `@RestController` in the application and converts them into a structured JSON error response.

> Reference: [Spring @RestControllerAdvice](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-advice.html)

Without this handler, Spring would return its default error HTML or a generic JSON with limited detail. With it, every error returns the standardized `ErrorResponse` shape.

### The 5 handlers

**1. `ApiException` — our application errors**

```java
@ExceptionHandler(ApiException.class)
public ResponseEntity<ErrorResponse> handleApiException(ApiException ex, HttpServletRequest request)
```

`ApiException` is a `RuntimeException` with an `HttpStatus` field. Any service can throw it:
```java
throw new ApiException(HttpStatus.NOT_FOUND, "User not found");
```
The handler wraps it in `ErrorResponse` with the correct status code.

**2. `MethodArgumentNotValidException` — input validation failures**

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ErrorResponse> handleValidation(...)
```

Thrown by Spring when `@Valid` fails on a request body. The handler collects all field errors and joins them with `"; "`:

```
"email: must not be blank; password: size must be between 8 and 100"
```

Returns `400 Bad Request`.

**3. `AccessDeniedException` — authorization failures**

```java
@ExceptionHandler(AccessDeniedException.class)
public ResponseEntity<ErrorResponse> handleAccessDenied(...)
```

Thrown by Spring Security when `@PreAuthorize` fails (e.g., a LAB_ANALYST calls a LAB_ADMIN endpoint). Returns `403 Forbidden`.

**Why does this need its own handler?**

`AccessDeniedException` extends `RuntimeException`. Without an explicit handler for it, it would fall through to the generic `Exception.class` handler below and return `500 Internal Server Error` — which is wrong (it's a 403, not a bug). The explicit handler intercepts it before the catch-all fires.

**4. `AuthenticationException` — authentication failures from Spring Security**

```java
@ExceptionHandler(AuthenticationException.class)
public ResponseEntity<ErrorResponse> handleAuthenticationException(...)
```

Thrown by Spring Security's internals (e.g., `UsernameNotFoundException`). Returns `401 Unauthorized`.

**5. `Exception.class` — unexpected errors**

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<ErrorResponse> handleGeneric(Exception ex, HttpServletRequest request)
```

The catch-all. Logs the full stack trace at `ERROR` level with the request URI. Returns `500 Internal Server Error` with a generic message ("An unexpected error occurred") — no internal details are leaked to the client.

### `ErrorResponse` record

**File:** `common/exception/ErrorResponse.java`

```java
public record ErrorResponse(
        Instant timestamp,
        int status,
        String error,
        String message,
        String path,
        String correlationId
) {
    public static ErrorResponse of(int status, String error, String message, String path) {
        return new ErrorResponse(
                Instant.now(), status, error, message, path,
                MDC.get("correlationId")  // automatically picks up the current request's ID
        );
    }
}
```

The `correlationId` is pulled from MDC automatically — no need to pass it as a parameter. Since `CorrelationIdFilter` always sets the MDC key before the request reaches the controller, `MDC.get("correlationId")` always returns the correct ID.
